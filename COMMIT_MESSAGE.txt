feat: Add TTS Provider Selection Dialog and Remove Automatic Fallback Logic

This commit introduces a comprehensive provider selection system that gives users
explicit control over TTS provider choice, replacing automatic fallback behavior
with clear availability status and user-driven selection.

## Major Features

### 1. Provider Selection Dialog
- New modal dialog (`src/ui/dialogs/provider_selection_dialog.py`) for selecting
  and testing TTS providers
- Real-time status checking for all providers (async, non-blocking)
- "Test All Providers" functionality that generates actual audio samples
- Visual status indicators (üü¢ Active, üî¥ Unavailable, üü° Testing)
- Shows provider details (name, version, type, description) when selected
- Supports all three providers:
  - Edge TTS 7.2.3 (current version)
  - Edge TTS 7.2.0 (Working) - Hugging Face method
  - pyttsx3 (Offline)

### 2. UI Integration
- Replaced provider combo boxes with "Select Provider..." buttons in:
  - TTS View (`src/ui/views/tts_view.py`)
  - Full Auto View - Add Queue Dialog (`src/ui/views/full_auto_view.py`)
- Added status indicators (green/red dots) next to provider names
- Provider availability checks before voice loading
- Voice selection disabled with clear message when provider unavailable:
  "Sorry, the provider you are trying to use is currently unavailable"

### 3. Removed Automatic Fallback Logic
- **Breaking Change**: When a provider is explicitly specified, the system
  no longer automatically falls back to other providers if the selected one fails
- TTS Engine (`src/tts/tts_engine.py`) now:
  - Checks provider availability before attempting conversion
  - Uses provider directly (no fallback) when provider is specified
  - Only uses fallback logic when no provider is specified (backward compatibility)
- Users must explicitly choose a provider based on visible status
- Clear error messages when selected provider is unavailable

## Technical Changes

### Core Logic
- `TTSEngine.convert_text_to_speech()`:
  - Validates provider availability when provider parameter is provided
  - Calls provider directly instead of `convert_with_fallback()` when specified
  - Returns False immediately if provider is unavailable (no silent fallback)
  
### UI Components
- `TTSView` and `AddQueueDialog`:
  - Check provider availability before loading voices
  - Disable voice combo box when provider unavailable
  - Show appropriate error messages
  - Update status indicators based on provider availability

### Threading
- `ProviderStatusThread`: Async status checking for providers
- `ProviderTestThread`: Async audio generation testing for providers
- Both use QThread for non-blocking UI operations

## Breaking Changes

automatically fall back to other providers if the selected provider fails.**

- Users must select an available provider based on visible status
- If selected provider becomes unavailable, conversion will fail with clear error

## Benefits

1. **User Control**: Users explicitly choose providers based on real-time status
2. **Transparency**: Clear visual feedback about what works and what doesn't
3. **No Silent Failures**: Provider issues are immediately visible
4. **Better Testing**: Users can test providers before using them
5. **Predictable Behavior**: Selected provider is used exactly as chosen

## Migration Notes

For existing code that specifies providers:
- Ensure provider availability is checked before use
- Handle provider unavailability gracefully
- Use the new dialog for provider selection in UI
- Consider removing automatic fallback expectations from tests
